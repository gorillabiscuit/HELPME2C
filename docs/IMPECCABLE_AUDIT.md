## Detailed Findings by Severity

### [P1] Hardcoded `slate-N` colour values across the system

- **Location**: dashboard-home.tsx (17 usages), title-detail page (14), rec-card-actions.tsx (5), settings/account page (6), group-member-list.tsx (4). 46+ across 5 inspected files; almost certainly more in unscanned components.
- **Category**: Theming
- **Impact**: DESIGN.md explicitly says "Default to the cool-slate role tokens (`muted-foreground` for secondary text, `border` for hairlines). Don't reach for `slate-N` numbered variants except when specifically needing the granular ramp." The current code violates its own design system. Consequence: a future rebranding pass would not propagate via tokens; every hardcoded `slate-{N}` site would need a hand edit. Also breaks dark-mode support if it ever lands (token-driven colours flip automatically; numerical variants don't).
- **Recommendation**: do a single targeted pass migrating the high-frequency mappings:
  - `text-slate-500` → `text-muted-foreground`
  - `text-slate-600` → `text-muted-foreground` (currently the same role, used for body+secondary text — fine to collapse)
  - `text-slate-700` / `text-slate-900` → `text-foreground`
  - `border-slate-200` → `border-border`
  - `border-slate-300` (dashed empty-state) → `border-border` or new `border-muted` token
  - `bg-slate-50` → `bg-muted`
  - `bg-slate-100` (chip background) → `bg-muted` or new `bg-chip` token
  - `bg-slate-900` → `bg-primary`
  - `text-white` (on `bg-slate-900` button) → `text-primary-foreground`
- **Suggested command**: `impeccable harden` (a hardening pass on theming consistency) or `impeccable polish` (final pass after).

### [P2] Touch targets <44×44px on rec-card actions

- **Location**: `apps/web/src/components/rec-card-actions.tsx:62-101` — the three buttons (Seen it / Hide / Rate) use `px-2 py-1 text-xs`. Computed: ~24px tall × ~50px wide. Same pattern in the right-column dashboard nav links.
- **Category**: Accessibility / Responsive
- **Impact**: WCAG 2.5.5 (AAA) recommends 44×44 CSS px touch targets; WCAG 2.5.8 (AA) recommends 24×24 minimum. Current rec-card buttons technically pass AA but are unpleasant to hit on mobile, especially three-in-a-row in a 2-column grid where each card is ~50% viewport width. The "Hide" and "Seen it" buttons are intentionally subtle, but accidental rate clicks via fat-finger remain plausible.
- **Recommendation**: bump to `px-3 py-1.5` or `h-8` minimum. Don't grow the visible weight; just expand the hit area via padding.
- **Suggested command**: `impeccable polish` (small surface adjustment).

### [P2] Heading hierarchy skip on dashboard

- **Location**: `apps/web/src/components/dashboard-home.tsx:74` (`<h1>` greeting) → :146 (`<h3>` per rec card). No `<h2>` for the rec-list section.
- **Category**: Accessibility (WCAG 1.3.1 Info and Relationships)
- **Impact**: Screen readers announce headings as a navigable outline; skipping `<h2>` makes the implied structure ambiguous ("what is the relationship between the greeting and the rec items?"). Low-traffic page so user impact is bounded.
- **Recommendation**: either drop the rec items to `<h3>` and add an `<h2>` "Recommendations" before the grid, OR drop the rec items to `<h2>` and make the empty-state heading also `<h2>`.
- **Suggested command**: `impeccable harden` (a11y hardening) or directly fix in the next polish pass.

### [P2] Filter chip row could leak through reduced-motion preference

- **Location**: dashboard-home.tsx:97 — the `<div>` containing the "Filtering by" chips uses no `transition` so this is a non-issue. Marking as **non-issue**, included so future readers see it was checked.
- **Category**: N/A
- **Impact**: None.

### [P3] No `priority` hint on first-row rec poster images

- **Location**: dashboard-home.tsx:135-141 — every poster `<Image>` is lazy-loaded.
- **Category**: Performance
- **Impact**: The first 4 rec posters are above-the-fold on desktop. Marking them `priority` would let next/image preload them, reducing LCP slightly. Estimate: 50-100ms LCP improvement on cold load.
- **Recommendation**: pass an index into the map and set `priority={i < 4}`.
- **Suggested command**: `impeccable optimize`.

### [P3] Missing focus-visible ring on rec-card link

- **Location**: dashboard-home.tsx:132 — `<Link className="group block">` has no explicit `focus-visible:ring-*`.
- **Category**: Accessibility
- **Impact**: Tailwind doesn't auto-add focus rings to `<a>` elements. Keyboard users tab through with no visible indication of which rec card has focus. The image / title may have the implicit native browser focus ring, but it's narrow and easy to miss.
- **Recommendation**: add `focus-visible:outline-2 focus-visible:outline-ring focus-visible:outline-offset-2` (or equivalent) to the Link className.
- **Suggested command**: `impeccable harden` or `impeccable polish`.

## Patterns & Systemic Issues

1. **`slate-N` everywhere** — confirmed across 5 inspected files; 46+ instances. The migration to role tokens is the single highest-leverage theming improvement.
2. **Inline pill / chip pattern** repeated without primitive — streaming-provider chips on the title page, per-member score pills on group-recs, tag chips on title-detail, "Filtering by" chips on the dashboard. Each is 4-8 lines of inline Tailwind. Worth abstracting as a `<Chip>` or `<Pill>` primitive in `components/ui/` if a 4th instance lands. **Watch list, not yet a fix.**
3. **No max-touch-target consistency** — different pages use different padding for similar action buttons. A "small destructive action" on the rec-card differs in dimension from "small destructive" on the group page member list.

## Positive Findings

- **shadcn/ui primitives consistently applied** — Card, Button, Dialog, Input cover the surface area. New code reaches for the right primitive.
- **Anti-patterns clean** — five common AI tells (purple gradient, glassmorphism, gradient text, hero metrics, nested cards) are absent. The deliberate restraint is working.
- **Image optimization correct** — `next/image` with `sizes` and `fill` everywhere; no `<img>` hardcodes.
- **OKLCH colour tokens** — even though usage skews to numerical `slate-N`, the underlying token palette is OKLCH, ahead of most projects.
- **Per-component decorative-image alt text** is correctly empty (`alt=""`) rather than redundant noise.
- **Heading-level decisions are intentional** — the "tag pills below title" and "filter chips below copy" patterns are good IA, even if the heading hierarchy itself has a gap.
