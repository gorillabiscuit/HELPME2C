# ADR-0019: ORM (Drizzle)

**Status:** Accepted
**Date:** 2026-05-06
**Supersedes:** —

## What we chose

Drizzle ORM with the `drizzle-orm` runtime package and `drizzle-kit` CLI for migrations. Schema is defined in TypeScript files inside `packages/shared/src/db/schema/`; migrations are generated by `drizzle-kit generate` and applied by `drizzle-kit migrate` (or `push` in development). The Neon serverless driver (`@neondatabase/serverless`) is used as the underlying Postgres adapter.

## What we rejected

- **Prisma** — schema is defined in a separate `.prisma` file (its own DSL), requiring a code-generation step (`prisma generate`) on every schema change. Heavy generated client, historical performance quirks on serverless connections, and an extra language to maintain alongside TypeScript.
- **Kysely** — purely a query builder, not an ORM. No migration tooling; would require a separate migration tool (e.g. `umzug`). Good choice if you want raw SQL proximity; not the right trade-off for a solo dev building at MVP pace.
- **Raw SQL / `pg`** — maximum control and performance, but requires hand-written migration files and no compile-time type checking on queries. The risk of type drift between query results and TypeScript types is too high for this codebase.
- **MikroORM** — full-featured but complex; the learning curve and configuration overhead outweigh the benefit at Phase 1A scale.

## Why

Drizzle is TypeScript-first: the schema is a TypeScript file, not a separate DSL. This means the schema lives in the same language as everything else, benefits from the same IDE tooling, and requires no codegen step. Changes to the schema are immediately reflected in query types without an intermediate generation step.

The Neon serverless driver (`@neondatabase/serverless`) pairs directly with Drizzle's Neon adapter, which is the combination Neon themselves recommend and maintain. This avoids the connection-pool mismatch problems that can arise with traditional `pg` drivers in serverless/edge environments.

Drizzle's query API has two styles: a relational query API (`db.query.users.findMany(...)`) that reads like TypeScript and a SQL-like builder (`db.select().from(users).where(...)`) for cases where you want fine-grained control. Having both available means we're never forced into either "pure ORM magic" or "write raw SQL strings."

For Phase 1A, the priority is velocity and correctness; Drizzle's TypeScript-native approach gives both without the operational overhead of Prisma's codegen pipeline.

## What would change our mind

- Drizzle's migration tooling proves insufficient for a complex schema change (large renames, data migrations). We'd evaluate Prisma's `prisma migrate` or a dedicated migration tool like `node-pg-migrate` at that point.
- The team grows beyond a solo dev and the SQL-like query style becomes a communication barrier compared to Prisma's more ORM-idiomatic API.
- A Drizzle-unsupported Postgres feature becomes critical (unlikely given Drizzle's active development, but worth noting).
